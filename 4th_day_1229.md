## 브랜치 Branch
- *독립적인* 작업흐름을 만들고 관리
- 작업 시작하면서 만들기
    - main branch에는 작업하지 않는게 국룰
    - 다른 branch에 작업 후 merge
- branch naming 전략이 있다.
  
### 브랜치 명령어
- `git branch {branch name}` : 해당 브랜치 생성 
- `git switch/checkout {branch name}` : 해당 브랜치로 이동
- `git checkout -b (또는 switch -c) {branch name} : 해당 브랜치 생성 후 이동
- `git branch -d {branch name}` : 해당 브랜치 삭제
- `git push {repo name} {branch name}` : 특정 branch만 push하기

## 병합 Merge
- main 브랜치로 병합시키려면 main 브랜치로 이동 후 `git merge`
- conflict 발생한 merge를 해결 후 `git commit` 
  
> ## Git Flow
> Git을 활용하여 협업하는 흐름, branch를 활용하는 전략.  
> 커밋은 저장이 아니라 개별적인 버전을 나타낸다.  


## Pull Request
Collaborator로 초대 -> 메일로 알람이 온다  
### 흐름
1. 초대수락
2. 클론
3. 브랜치 만들기
4. 작업후 `git push origin {branch name}

## 초대 받지 않았을 때 : Fork
### 흐름
1. 해당 GitHub 저장소를 fork한다
2. fork 된 **자신의** 저장소를 로컬저장소로 clone하여 작업한다.
    - 브랜치 만들기 필수
3. add, commit, push 후 pull request한다.
4. Pull request가 수락되면 원래의 저장소에 반영된다.

## Restore
add된 파일을 add되지 않은 상태로 돌리거나, 아직 add되지는 않았으나 커밋후 수정된 내용을 수정 이전의 상태, 즉 커밋 된 시점의 상태로 복구(Restore)하기 위해 사용.  
- add된 파일 복구 :  
> `git restore --staged <file> ...`  
> 당황하지 않고 Torvals 아저씨의 안내를 따라간다.
- 커밋 된 시점의 상태로 되돌리기 : 
> `git restore <file> ...`

## 커밋 메세지 Commit Message
커밋 메세지를 수정할 수 있다 : `git commit --amend`  
-> 새로운 텍스트 창이 열린다  
> 수정시 해쉬값이 달라진다.  

뭔가를 되돌리는 행위는 로컬에서는 괜찮지만(그래도 신중하라) 원격저장소로 푸쉬 한 이후에는 곤란하다. 같이 일하는 다른 사람들을 배려하라.  
일관성과 회사의 규칙 등 스타일 가이드를 좇아라.
